# Quality Gate Redesign + Auto-Correct + Conversational Revision

## Problem

The system already has rules that catch AI tropes, voice mismatch, recapping conclusions, etc. But:
1. The quality gate runs AFTER generation and only shows warnings — it never fixes them
2. The user sees a draft full of issues the system already knows about, then has to leave the platform (e.g. paste into Claude) to iterate on framing and perspective
3. The quality gate shows what's WRONG instead of surfacing what needs HUMAN JUDGMENT

## Solution: Three Connected Changes

### 1. Auto-Correct Before the User Sees the Draft

**Flow**: Generate drafts → combine → run through coach-check prompt → show user the cleaned version

The coach-check prompt receives:
- The combined draft
- All writing rules (voice/tone, structure/formatting, anti-AI tropes)
- All active coaching insights
- The 6 quality dimensions (voice match, AI tropes, hook strength, engagement close, concrete specifics, ending quality)

It evaluates the draft against all of these and rewrites to fix any violations in a single pass. The user never sees the intermediate version — they get the best version the system can produce.

This is NOT a separate step from the quality gate. It replaces the current quality-gate-as-scorer pattern. The quality gate becomes the coach-check-and-fix step.

**Implementation**:
- New module (e.g. `coach-check.ts`) that takes a draft + rules + insights and returns the cleaned draft + the two assessment sections (see below)
- Called in the combine endpoint after `combineDrafts()`, before returning to the user
- Also called after each revision in the conversational flow
- The prompt should instruct the model to fix issues silently and produce the best possible version, not explain what it changed

### 2. Quality Gate Becomes Two Sections

The quality gate output changes from pass/warn violation checks to:

**Section 1 (top, prominent): "Needs Your Expertise"**

Things the AI cannot resolve with rules alone. These are framing choices, perspective decisions, and domain knowledge gaps. Examples from the Delve post scenario:
- "This frames the audit failure as enterprises not verifying their auditors — is that the right framing, or is the structural trust model itself the point?"
- "The post attributes the root cause to automation/price pressure — does that match what you've seen driving this?"
- "The closing targets vendor assurance program owners — is that your intended audience?"
- "There's a first-person anecdote about a vendor review meeting — does this match a real experience you can draw from, or should we find a different moment?"

These are generated by the coach-check step. The model should identify 2-4 areas where the user's actual knowledge and perspective would change the post. This is what kicks off the user's iteration — not fixing AI mistakes, but applying their expertise.

**Section 2 (bottom, secondary): "Why This Aligns"**

Positive confirmation that the rules were followed:
- Voice: practitioner tone throughout, no analyst-mode passages
- Hook: opens with friction/claim, no context dump
- Tropes: no banned words, no correlative constructions
- Close: practitioner process question, not generic opinion ask
- Specifics: named firms, concrete failure modes, real systems referenced
- Ending: extends the idea forward, no recap

This is a confidence builder. The user sees green checks confirming the system did its job.

**Data model change**: The `quality_gate_json` column on `generations` changes shape from:
```json
{
  "passed": true,
  "checks": [{ "name": "...", "status": "pass|warn", "detail": "..." }]
}
```
to:
```json
{
  "expertise_needed": [
    { "area": "string — short label", "question": "string — what the user should weigh in on" }
  ],
  "alignment": [
    { "dimension": "voice_match|ai_tropes|hook_strength|engagement_close|concrete_specifics|ending_quality", "summary": "string — why this dimension is satisfied" }
  ]
}
```

### 3. Conversational Revision

Once the user sees the cleaned draft + "needs your expertise" prompts, they iterate through conversation rather than single-shot action buttons.

**Data model**:
```sql
CREATE TABLE generation_messages (
  id INTEGER PRIMARY KEY,
  generation_id INTEGER REFERENCES generations(id),
  role TEXT NOT NULL, -- 'user' | 'assistant'
  content TEXT NOT NULL,
  draft_snapshot TEXT, -- full draft state after this message (assistant messages)
  quality_json TEXT, -- expertise_needed + alignment after revision (assistant messages)
  created_at DATETIME DEFAULT CURRENT_TIMESTAMP
);
```

**API**: `POST /api/generate/chat`
- Takes `generation_id` and `message` (user's feedback/instruction)
- Optionally takes `edited_draft` if the user made inline edits before sending
- Loads full conversation history for this generation
- Sends to model with system prompt (writing rules + coaching insights) + conversation history
- Model makes targeted changes based on user feedback, returns revised draft
- Runs the coach-check on the revised draft (auto-fixes any new rule violations introduced)
- Returns: revised draft, new expertise_needed/alignment assessment, explanation of changes

**System prompt for conversation mode**:
- Include all writing rules and coaching insights
- Instruct the model to make targeted changes, not full rewrites, unless the user asks
- When the user gives framing/perspective feedback, apply it and briefly explain what changed
- If the user's feedback is ambiguous, ask one clarifying question before rewriting

**Existing action buttons become pre-filled messages**: "Shorten", "Strengthen close" send their current prompt text through the chat endpoint. They're convenience shortcuts, not a separate code path.

**The "needs your expertise" prompts become clickable conversation starters**: When the user clicks one, it pre-fills the chat with that question so they can respond with their perspective. The AI then revises based on their answer.

### Frontend Changes

**ReviewEdit component** evolves:
- Left: editable draft (inline editing still works)
- Right: conversation thread
- Top of right panel: "needs your expertise" cards (clickable to start conversation)
- Below: chat messages (user instructions + AI responses with targeted revisions)
- Bottom of right panel: "alignment" section showing green checks
- Action buttons (shorten, strengthen close) become quick-reply chips in the chat

## Implementation Order

1. **Coach-check module** (`coach-check.ts`) — the prompt that evaluates + fixes + produces the two assessment sections
2. **Wire into combine endpoint** — replace current `runQualityGate` call with coach-check
3. **Update quality gate JSON shape** — new schema for expertise_needed + alignment
4. **Frontend: update quality gate display** — show new two-section layout instead of pass/warn checks
5. **Database: add generation_messages table**
6. **API: add chat endpoint** — conversational revision with history
7. **Frontend: add chat panel** — conversation UI alongside draft editor
8. **Wire expertise prompts as conversation starters** — clickable cards that pre-fill chat
9. **Migrate action buttons to chat** — shorten/strengthen become pre-filled messages

## Key Prompt Design Notes

The coach-check prompt is the most critical piece. It needs to:
- Fix rule violations without being told what they are (it reads the rules and applies them)
- NOT over-edit — preserve the argument structure and specific content
- Identify areas needing human expertise — this requires understanding what's a RULE issue (fixable) vs a FRAMING issue (needs the author)
- Produce positive alignment confirmation — not just absence of violations, but specific reasons why each dimension is satisfied

The distinction between "fixable by rules" and "needs human judgment" is the core intelligence of this system. Rule violations (banned words, correlative constructions, recap paragraphs) get fixed silently. Framing choices (who's to blame, what's the root cause, what audience to target, whether an anecdote is authentic) get surfaced as questions.
